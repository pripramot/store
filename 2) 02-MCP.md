# Module: MCP — Model Context Protocol Gateway (RBAC + Audit)
Owner: System Architect / Security
Scope: MCP Server as the ONLY gate to DB & evidence services (local-first)

---

## ✅ Copilot Prompt (Paste into Copilot)
You are **Phuphadang Alpha AI (Forensics Edition)** acting as a system architect and security engineer.

### Goal
Design and draft a **MCP Server** (Go or Node.js) that mediates all access to evidence DB/services using **RBAC**, **least privilege**, and **forensic-grade audit logging**.

### Context
- Local-first / on-prem only.
- Agents must not query DB directly.
- Every request must be logged with who/when/what/why + payload hash.
- No hallucination: if something is unknown, state assumptions clearly.

### Source of truth (Inputs)
- Roles: `Investigator`, `Analyst`, `Admin`, `Auditor`
- Resources: `evidence`, `hash_ledger`, `signature_ledger`, `audit_log`, `case`
- Required audit fields: `request_id`, `actor_id`, `role`, `action`, `resource`, `timestamp`, `reason`, `result`, `payload_hash`, `prev_audit_hash`, `audit_hash`

### Expectations (Output)
Return in Markdown:
1) Text architecture diagram (components + trust boundaries)
2) RBAC matrix (role → allowed actions)
3) Endpoint list (OpenAPI-ish)
4) Reference code skeleton (choose Go OR Node.js; justify)
   - auth (mTLS or signed JWT locally)
   - authorization middleware (RBAC)
   - audit writer (append-only hash chain)
5) DB schema for audit log (if not already defined)
6) Security notes: key management, rotation, tamper detection, offline time sync

### Non-negotiable Controls
- Deny-by-default policy
- All writes require `reason` field
- All reads must be scoped by `case_id` and logged
- Audit log is tamper-evident (hash chain)
- Support "break-glass" access with extra audit markers

Now produce the deliverables.

---

## RBAC Baseline (Suggested)
- Investigator: ingest, view case-scoped evidence metadata, export with approval marker
- Analyst: run analysis jobs, read evidence copies, cannot delete
- Admin: manage users/roles/keys, cannot alter evidence content
- Auditor: read-only audit logs + verification tools

---

## Acceptance Criteria
- Every MCP request produces an audit record (including denied requests)
- Unauthorized access returns consistent errors without leaking data
- Role changes are auditable and require admin + reason
- Tampering with audit rows is detectable by hash chain verification
